## 执行步骤

### 1. 扫描当前 Phase
- 定位当前 Phase 目录（由上下文或用户指定）。
- 用 `Glob` 找出所有 `*.done.md` 和 `*.blocked.md` 任务文件。
- `Read` 这些文件，提取：
  - 完成摘要中的**设计模式 / 架构技巧**
  - 审查意见中**被多次打回的问题类型**（同一 Task 累计 retry ≥ 2）
  - 测试日志中**新增的边界条件或 Mock 策略**
  - 计划文件中记录的**技术选型理由**

### 2. 提炼通用经验（必须跨项目抽象）
所有提炼结果必须满足：
- **不包含**当前项目的 Task ID、文件路径、具体接口名称。
- 使用通用术语描述（例如不说 `TASK-003`，而说“速率限制中间件的责任链模式”）。
- 按以下维度整理：

| 维度 | 说明 |
|:---|:---|
| **模式** | 可复用的设计模式、代码结构、架构分层 |
| **反模式 / 常见陷阱** | 导致审查或测试多次失败的典型错误模式 |
| **决策** | 技术选型理由（库、框架、协议等）及权衡 |
| **测试策略** | 有效的测试方法、边界条件构造技巧、Mock 方案 |
| **审查清单** | Reviewer 高频关注点，可直接作为后续开发预防项 |

### 3. 写入 `KNOWLEDGE.md`
- 若文件不存在则新建，编码 UTF-8。
- 追加内容格式：
  ```
  ## [YYYY-MM-DD] Phase-XX 名称
  ### 模式
  - ...
  ### 反模式 / 常见陷阱
  - ...
  ### 决策
  - ...
  ### 测试策略
  - ...
  ### 审查清单
  - ...
  ```
- 只写抽象经验，不贴具体代码；如需示例，用伪代码或最小化片段。
- 每条经验开头用一句话概括，让读者 5 秒内能判断是否相关。

### 4. 存入知识图谱（mcp__memory__create_entities）
- 为每条提炼出的通用经验调用 `mcp__memory__create_entities`。
- **实体名称**必须抽象，例如：
  - `错误处理反模式-重复try-catch`
  - `测试策略-外部API模拟`
  - `技术选型-JWT库选择`
  - `审查清单-SQL注入防护`
- 实体 `description` 字段写入经验的简短描述（≤300 字）。
- 实体 `tags` 字段至少包含对应维度（如 `#反模式`）和适用场景（如 `#后端` `#异步`）。
- 如果两个实体存在关联（如某反模式源于某决策），可以建立关系，但非必须。

### 5. 完成报告
- 汇总本次提炼的维度、实体数量、写入 `KNOWLEDGE.md` 的行数。
- 输出简短总结（无需调用其他 Agent）。
