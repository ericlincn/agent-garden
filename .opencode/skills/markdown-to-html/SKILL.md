## 前置条件

用户提供一个 Markdown 文件的路径或内容，例如：
- `./docs/prd-产品名-20250101.md`
- `./reports/analysis-主题-20250101.md`
- `./KNOWLEDGE.md`
- 或直接粘贴 Markdown 文本

**目标**：产出 `{原文件名}.html`，一个包含 CSS/JS/HTML 的独立文件。

## 核心定位

这不是简单的 `.md → .html` 转换器，而是一个 **杂志级排版引擎**。理解原文档的信息结构，重新组织为易读、美观、有层次的布局：

- 保留原文档的**完整信息**，不删改任何实质性内容
- 增强**可读性**：通过字体层级、间距、颜色、卡片、图标让信息一目了然
- 增强**可探索性**：生成目录、锚点链接、折叠区块、筛选/排序表格等交互
- **智能识别**：自动检测文档类型（PRD、行业分析、知识沉淀、通用），采用对应的排版模板

## 执行流程

### 阶段 0：读取与类型识别

1. 读取 Markdown 文件内容（若用户提供路径，使用 Read 工具；若直接粘贴，使用给定内容）
2. 分析文档特征，识别类型：

| 文档类型 | 识别特征（满足任一） | 特殊排版侧重 |
|---------|---------------------|-------------|
| **产品策略与需求文档 (PRD)** | 标题含“产品策略与需求文档”“PRD”；章节有“发现机会”“需求拆解”“技术规范”“MVP交互原型包” | 表格 → 交互式数据表；用例编号 → 徽章；技术规范 → 代码块美化；场景矩阵 → 卡片网格 |
| **交叉分析报告** | 标题含“交叉分析报告”；章节有“维度”“矛盾与交叉验证”“综合结论”；有“核心发现”“信心水平”等 | 维度 → 带图标的时间线/卡片；证据引用 → 脚注/引用块；信心水平 → 进度条或标签 |
| **知识沉淀 (KNOWLEDGE.md)** | 文件名含 KNOWLEDGE；章节有“模式”“反模式”“决策”“测试策略”“审查清单” | 每条经验 → 独立卡片；标签系统；可折叠详情 |
| **通用** | 以上都不匹配 | 标准文章式排版，但同样应用杂志风格 |

3. 若无法自动确定，询问用户选择类型，但尽量自动推断。

### 阶段 1：风格选择

向用户展示可选的风格主题，让用户选择其一（也可默认第一种）：

1. **优雅编辑** (`editorial`)
   - 衬线标题 + 无衬线正文，暖灰背景，低饱和度强调色，宽边距，适合长文深度阅读
2. **科技蓝调** (`tech`)
   - 深蓝/青色系，半透明面板，等宽字体用于数据，发光边框，适合 PRD 与技术规范
3. **暗黑炫彩** (`dark`)
   - 深色背景，高饱和霓虹点缀，卡片浮起感，适合行业分析报告、演示场景
4. **柔和自然** (`nature`)
   - 米色/浅绿，圆润边框，柔和阴影，舒适大行距，适合知识沉淀与复盘文档
5. **活力杂志** (`magazine`)
   - 高对比撞色，粗标题，多栏布局（桌面端），图片占位符装饰，适合营销/产品故事

> 若用户未指定，默认使用“优雅编辑”。所有风格都支持响应式（移动端自动调整为单栏）。

### 阶段 2：解析与结构化

使用 `marked` 或自定义解析器提取以下元素，构建中间表示（JSON）：

- 标题层级（H1–H6）及层级关系（目录树）
- 段落、列表（有序/无序）、嵌套列表
- 表格（表头、行、单元格）
- 代码块（含语言标记）
- 引用块
- 图片引用（`![alt](url)`）
- 粗体/斜体/行内代码等内联标记
- 特殊标记检测：`[代码实证]`、`[团队口述]`、`[AI 推断]`、`UC-01` 等用例编号、`Phase 1`/`Task-xxx` 等任务标记

同时记录原始 Markdown 的完整内容，确保无一遗漏。

### 阶段 3：智能排版与组件化

根据文档类型和选择风格，将中间表示渲染为 HTML 组件。以下为各类型的处理规则：

#### 全局组件（所有文档）

- **目录**：生成侧边栏抽屉（桌面端固定右侧或左侧，移动端可隐藏按钮）。点击锚点平滑滚动。
- **标题装饰**：H1 采用杂志大标题 + 细线装饰；H2 带小图标或背景色块；H3 下划线或左框线。
- **代码块**：使用 Prism.js 或 highlight.js 进行语法高亮，显示行号和语言标签。
- **表格**：转换为 `<div class="data-table">` 包装的 `<table>`，添加排序功能（点击表头排序）和搜索输入框（针对表格内容）。若表格有数值列，自动绘制微型 sparkline 或柱状条（使用 CSS 渐变）。
- **列表**：超过 5 项的列表自动转为网格卡片（每项一个卡片），少于 5 项的保持传统列表但添加装饰图标。
- **引用块**：设计为侧边大引号 + 浅色背景，作者/来源单独样式。

#### PRD 文档专属

- **Phase / Task 分组**：识别 `Phase X：` 标题，渲染为时间线组件，每个 Task 显示为带优先级徽章（Must/Should/Could）和工时标签的卡片。
- **场景矩阵**：如果表格内容包含“角色 × 状态 × 数据状态”，自动生成矩阵可视化（使用 CSS Grid 绘制带颜色块的状态图）。
- **API 端点列表**：将 `/api/xxx` 格式的表格转换为类似于 Postman 的可折叠端点卡片，展示方法（GET/POST）、路径、描述。
- **数据模型 ER 图**：如果有实体关系文本描述，生成基于 Mermaid 的 ER 图（嵌入 Mermaid.js）。
- **MVP 交互原型包**：如果章节包含文件清单，生成可下载的模拟文件树（不可实际下载，仅展示结构）。

#### 交叉分析报告专属

- **维度**：每个维度渲染为独立的“杂志文章”区块，带维度编号大标识。内部结构：“核心发现”作为 pull quote，“详细分析”分栏（两栏桌面），“关键证据”转换为带来源标签的引用列表。
- **信心水平**：显示为进度条（高=100%，中=65%，低=30%）或彩色徽章。
- **矛盾与交叉验证**：渲染为对比卡片，左右列出矛盾双方。
- **估值与风险报酬比**：如果有 DCF 表格或参数表，自动生成可交互的敏感性分析滑块（通过 JavaScript 调整增长率等参数动态更新估值结果）。

#### 知识沉淀专属

- **模式 / 反模式 / 决策**：每个条目生成一张卡片，卡片标题为大写小标签（PATTERN / ANTI-PATTERN / DECISION）。卡片底部添加“适用场景”标签（从 tags 字段提取）。
- **审查清单**：转换为可勾选的 checkbox 列表，支持本地存储保存勾选状态。
- **时间线**：根据日期（`[YYYY-MM-DD]`）自动排序，生成时间轴视图。

#### 通用文档

- 标准文章式布局：最大宽度 960px 居中，正文 18px，行距 1.6，图文混排。
- 自动生成“阅读进度条”顶部。

### 阶段 4：增强交互与图表化

在不丢失信息的前提下，将部分文本数据转换为图表：

- **表格数值列**：如果表格有数值且行数 ≥ 3，在表格顶部添加“图表视图”按钮，点击后使用 Chart.js 渲染为条形图/折线图（根据表头选择）。
- **列表中的键值对**：如果列表项呈现“指标：数值”模式，提取为仪表盘样式的小标尺。
- **时间序列数据**：识别日期列（如 2024-01, 2024-02）和数值列，自动绘制趋势线。

所有图表数据来源于原文档，不引入外部数据文件。

### 阶段 5：生成 HTML 文件

将上述所有组件整合到一个完整的 HTML 文件中，遵循以下要求：

- **单文件**：所有 CSS、JavaScript 内联或嵌入（外部库通过 CDN 链接，但需要明确版本和 integrity hash，优先考虑轻量级库）。
- **依赖最小化**：默认使用原生 JavaScript，除非复杂图表（Chart.js）或语法高亮（Prism.js）才引入 CDN。总 CSS 不超过 200 行定制，JS 不超过 300 行。
- **加载速度**：使用 `defer` 或异步加载非关键 JS，首屏只渲染静态内容。
- **响应式**：移动端适配（<768px 单栏，>1200px 多栏或固定最大宽度）。
- **无需外部服务器**：所有链接为相对路径或 `#`，图片若为网络 URL 直接引用，若为本地路径则保留并提醒用户自行托管。
- **元信息**：在 `<head>` 中添加原 Markdown 文件名、生成时间、转换引擎版本。

### 阶段 6：交付与后处理

将生成的 HTML 保存到与原 Markdown 相同目录下，文件名为 `{原文件名}.html`。若用户未指定输出路径，保存到 `./converted/` 目录（自动创建）。

展示预览信息：文件大小、风格主题、文档类型、交互特性清单（可排序表格、图表、折叠块等）。

提供可选后续操作：
- 是否需要调整某些元素的样式（如配色、字体大小）
- 是否需要添加“打印优化”样式（@media print）
- 是否需要生成暗色/亮色模式切换按钮（若当前风格未包含）

## 设计系统原则（来自 Web Design Engineer skill）

- **避免 AI 风格化**：不要过度使用紫色-粉色渐变、左侧边框圆角卡片、Emoji 作为图标替代。
- **使用占位符**：缺失图标时使用简单几何图形 + 文字 `[图标]`，不要编造。
- **大胆留白**：标题与正文的比例可达 1:4，行距 1.5–1.8。
- **精致的点缀**：利用 `::before` / `::after` 绘制装饰线、斜条纹背景、浅色水印等。
- **字体选择**：优先考虑品牌字体（若文档中有），否则从推荐列表中选（Plus Jakarta Sans, Sora, Newsreader, Space Grotesk），**避免 Inter / Roboto / Arial**。
- **色系统一**：使用 oklch 定义主色和辅助色，生成和谐色板。

## 技术实现参考

推荐使用以下库（通过 CDN 引入，注意版本锁定）：

```html
<!-- 解析 Markdown（可选，如果需要在前端解析，但模型已解析好，可以直接输出结构）-->
<!-- 实际上模型应该在后端完成解析，直接输出 HTML，不需要在前端解析 Markdown -->
<!-- 但为了灵活性，也可以将 Markdown 原文作为 data 属性嵌入，由前端 marked.js 渲染 -->
<!-- 此处为了单文件可靠性，推荐模型直接输出静态 HTML（已渲染内容）。-->
```

**模型直接生成渲染后 HTML**：由于模型具备解析能力，应当在生成阶段就将 Markdown 转换为 HTML 标签，无需在浏览器端二次解析。这样确保内容准确且无依赖。

图表库：
```html
<script src="https://cdn.jsdelivr.net/npm/chart.js@4.4.0/dist/chart.umd.min.js"></script>
```

语法高亮：
```html
<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/prismjs@1.29.0/themes/prism-tomorrow.min.css">
<script src="https://cdn.jsdelivr.net/npm/prismjs@1.29.0/prism.min.js"></script>
```

**注意**：为了单文件，模型应尽量内联 CSS 和简单的 JS，仅对确实需要的功能引用外部库，并检查 CDN 可用性。

## 输出模板结构

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0, viewport-fit=cover">
    <title>{文档标题} - 杂志排版</title>
    <style>
        /* 设计令牌 (CSS 变量) */
        /* 全局样式 */
        /* 杂志组件样式（卡片、表格、目录等） */
        /* 响应式 */
        /* 打印样式 */
    </style>
</head>
<body>
    <div class="magazine-wrapper">
        <aside class="toc" aria-label="目录">...</aside>
        <main class="magazine-content">
            <header class="magazine-header">...</header>
            <article>
                <!-- 渲染后的内容 -->
            </article>
            <footer class="magazine-footer">
                <p>生成时间：... | 来源：{原文件名}</p>
            </footer>
        </main>
    </div>
    <script>
        // 交互逻辑：表格排序、图表初始化、目录滚动、折叠等
    </script>
</body>
</html>
```

## 质量检查清单

- [ ] 原 Markdown 的所有文本内容均可见，无遗漏段落或表格
- [ ] 标题层级正确映射为 HTML 标题（h1-h6）
- [ ] 代码块语法高亮正常工作
- [ ] 表格排序与搜索可用
- [ ] 图表数据与原表格数据一致
- [ ] 目录链接可正确跳转至锚点
- [ ] 在 Chrome / Firefox / Safari 最新版上显示正常
- [ ] 移动端无水平滚动，字体足够大（≥16px）
- [ ] 无控制台错误
- [ ] 未使用过度 AI 风格特征（紫粉渐变、左框线卡片、无意义 emoji）
- [ ] 文件大小合理（外部库已最小化，CSS 压缩）

## 交互与迭代

生成 HTML 后，询问用户是否需要修改：

- 调整某一特定组件的样式（如表格的行高、卡片的圆角）
- 添加“深色模式切换”按钮
- 修改图表类型（如条形图改为折线图）
- 导出一个用于打印的版本（隐藏导航、调整页边距）

用户同意后，相应修改 HTML 并重新保存。

**最终产出**：一个 `.html` 文件，能够脱离任何编辑器或服务器，在浏览器中打开即呈现精致的杂志风格文档。
