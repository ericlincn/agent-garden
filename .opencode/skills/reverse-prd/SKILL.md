---

## 前置条件
- 当前工作区存在一个互联网产品（Web/App/游戏）的代码库。
- 用户能够提供业务定位等补充信息（若不懂代码逻辑，AI 会主动解释并引导确认）。

## 执行流程

### 步骤 1：深度代码扫描（不仅仅是路由和文件）
使用系列工具按以下层次全面扫描，并将发现写入临时笔记。

**1.1 项目宏观边界**
- 使用 `Glob` 获取项目根目录结构。
- 读取 `README.md`、`package.json` / `Cargo.toml` / `requirements.txt` 等，推断技术栈和项目名称。

**1.2 路由与入口**
- 使用 `Grep` 搜索路由定义（如 `app.get(`、`Router(`、`@app.route`、`createRouter`）。
- 提取全部 API 端点或页面路径列表，构建“用户可见功能点”初稿。

**1.3 数据模型深度还原**
- 搜索数据库模型定义文件（如 `models.py`、`schema.prisma`、`*.entity.ts`），读取每个模型的字段、类型、默认值、关联关系和注释。
- 推断核心实体及其业务含义（如 `Order.status` 有 `pending/paid/shipped/cancelled` 可推断订单流程）。
- 若存在迁移文件（`migrations`），读取建表 SQL，理解字段级别的历史演变。

**1.4 核心业务逻辑**
- 搜索关键服务层文件（如 `*service*`、`*usecase*`），读取核心函数/方法名与逻辑片段，理解主要业务流程。
- 搜索状态机、定时任务（`cron`、`schedule`）、消息队列消费者，挖掘隐式功能。

**1.5 交互与前端**
- 若前端代码存在，扫描组件树和状态管理，还原页面跳转关系和核心交互。
- 搜索任何八角行为相关关键词（积分、徽章、等级、排行榜）推断游戏化设计。

**1.6 配置与环境**
- 读取配置文件（`.env.example`、`config.py`），了解第三方服务（支付、推送、AI 等）和特性开关。

**[增强] 1.7 代码活跃度分析（过滤死代码/废弃逻辑）**
- **路由/端点活跃度**：
  - 检查每个路由是否被其他模块 `import` 或注册到主路由。使用 `Grep` 搜索路由处理函数的引用（函数名跨文件匹配）。
  - 若项目使用 Swagger/OpenAPI，比对文档与实际路由的差异，标记“文档有但代码无”为已废弃。
  - 若可访问 `git log`（当前目录为 git 仓库），提取每个路由对应的文件最后 commit 时间，超过 1 年未修改且无引用 -> `[疑似废弃]`。
- **前端组件活跃度**：
  - 检查组件是否被其他页面或组件 `import`（用 `Grep` 匹配组件名）。
  - 搜索 `TODO`、`FIXME`、`// deprecated` 注释，记录占位或待删除代码。
- **数据库表/字段活跃度**：
  - 若字段从未在代码中被 SELECT/UPDATE（通过 `Grep` 搜索字段名），且在迁移文件中被 `DROP COLUMN` 或注释 -> `[已废弃]`。
  - 标记所有未使用的模型和表，供后续过滤。
- **输出格式**：生成一份“代码事实清单”时，增加 `活跃状态` 列（活跃/疑似废弃/已废弃/不确定）。对于确定废弃的端点/组件，在逆向 PRD 中标记 `[已下线]` 并说明原因（如“代码存在但无任何引用”）。

> 此阶段对用户无打扰，AI 独自完成代码分析并形成一份“代码事实清单”。

### 步骤 2：用户补充与业务还原
根据代码事实清单，用 `AskUserQuestion` 逐个击破信息缺口，逐步还原你的产品设计框架的每一层。

**2.1 发现机会层**
- 告知用户从代码推断出的功能集合，然后问：*“这个系统的业务定位是什么？解决的是用户什么根本问题？”*
- 若能从代码中推断出目标用户（如面向 B 端有租户隔离），则确认并追问用户画像。
- 询问：*“当时为什么决定做这个产品？市场上主要竞品是谁？”* 帮助填充市场背景。
- 尝试还原“产品逻辑一句话”，如果用户说不清，AI 可根据代码功能草拟一句并请用户修正。

**2.2 需求拆解层**
- 展示从路由和模型还原的用户用例草稿，请用户补充或纠正。
- 展示根据模型和字段推断的功能列表，请用户标记哪些已废弃、哪些是核心。
- 询问：*“产品内部有过任何游戏化、激励、留存机制的考量吗？如果有，是基于什么用户动机设计的？”* 引导套出八角行为驱动因素。
- 若代码中有明显的数据关系，直接绘出 ER 草图，请用户确认业务规则（如一个订单能否包含多个商家商品）。

**2.3 资源与执行层**
- 询问团队构成、技术选型的原因、当时是否做过成本核算。
- 如果代码中有 MVP 痕迹（如大量 TODO、简化实现），请用户回忆当时的迭代策略和冷启动方法。

**2.4 风险与挑战层**
- 请用户回顾：*“产品运行过程中遇到过哪些致命风险？比如性能瓶颈、法律法规、用户流失等。”*

---

### 步骤 3：生成逆向 PRD
整合代码事实 + 用户补充信息，使用与 **produce-prd** 相同的文档模板输出到 `./docs/prd-reverse-{项目名}-YYYYMMDD.md`，但在文档开头增加：

```markdown
> **文档说明**：本文档基于代码逆向分析与团队回溯生成。部分商业背景和用户动机基于回忆和推断，可能存在偏差，已在各节标注“推理”或“待验证”标签。
```

文档内所有完全由代码支撑的结论标记为 `[代码实证]`，用户补充的标记为 `[团队口述]`，AI 推测的标记为 `[AI 推断]`。

**[增强] 同时，必须包含“章节映射与填充策略”及“置信度评分”**：  
在生成文档之前，AI 需按照以下强契约映射规则将扫描结果填入 produce-prd 的各子章节。若某章节无法从代码或用户处获取信息，执行规定的降级策略。

#### 3.1 章节映射与填充策略（强契约）

| produce-prd 章节 | 填充依据 | 缺失处理策略 | 标签强制要求 |
|----------------|---------|-------------|-------------|
| **1. 发现机会** | 用户口述 + 代码推断（如从命名推测行业） | 若用户未明确，保留占位符 `[待补充]` 并标注 `[AI 推断]`；不自动编造北极星指标 | `[团队口述]` / `[AI 推断]` |
| **2. 需求拆解** | 路由+模型+服务层映射，生成 UC；废弃功能标 `[已下线]` | 未找到任何用例时，生成 1 个通用 UC“用户访问系统”，标记 `[代码实证不足]` | `[代码实证]` / `[AI 推断]` |
| **3. 技术规范** | 配置文件+依赖包+数据库迁移 | 未找到的规范（如命名约定、文件夹结构）留空并标注“代码未体现规范” | `[代码实证]` / `[代码缺失]` |
| **3.7.5 API示例** | 路由定义+请求解析+Mock数据推断 | 若无反序列化逻辑，仅输出端点签名、请求方法、预期参数，附加 `[AI 推断]` 示例（需标注推测依据） | `[代码实证]` 或 `[AI 推断]` |
| **4. MVP原型包** | 前端组件树+路由跳转 | 无前端代码时跳过此章节，改为生成“接口调试清单”（含 curl 命令），标注 `[代码缺失]` | `[代码缺失]` |
| **5. 资源与执行** | 团队口述 + 配置中的第三方服务密钥 | 无信息时仅输出“未从代码或团队获取”，不生成虚假内容 | `[团队口述]` / `[代码实证]` |
| **6. 风险与挑战** | 用户口述 + 代码中明显的性能/安全缺陷（如 N+1 查询、硬编码密钥） | 若未发现任何风险，输出“代码扫描未发现明显风险，但未覆盖业务风险” | `[代码实证]` / `[团队口述]` |
| **7. 测试用例** | 状态机+边界条件推断（基于模型字段必填、枚举值） | 仅基于可验证的代码逻辑生成（如 `User.status` 枚举项导致的不同 UI），不编造业务规则；若无状态机则留空 | `[代码实证]` |
| **8. 附录** | 提取项目中的术语定义（常量文件注释）、外部文档链接 | 无则省略 | `[代码实证]` / `[代码缺失]` |

#### 3.2 降级与置信度规则
- **置信度评分**：为每个核心章节（1~7）输出一个置信度等级：`高`（>80%内容来自 `[代码实证]` 或 `[团队口述]`）、`中`（40%~80%）、`低`（<40%）。  
  在章节末尾以 `> **置信度**：[高/中/低] (依据：... )` 形式注明。
- **自动降级标记**：若某章节 >60% 内容依赖 `[AI 推断]`，自动在该章节顶部插入 `⚠️ 本章节代码实证覆盖率较低，建议优先验证`。
- **禁止行为**：对于 produce-prd 中明确要求“用户确认”的字段（如商业模式、北极星指标、成本估算），若用户未提供，禁止自动生成完整段落。必须保留占位符 `[待用户补充]` 并附上 `[AI 推断]` 草稿供用户确认。

#### 3.3 如何处理 API 示例（3.7.5）
- **优先读取**：若项目有测试文件（`*.test.js`、`*_test.py`）或 Postman 集合，直接提取其中使用的请求/响应示例。
- **其次推断**：根据路由定义（参数类型、返回类型）和模型字段，生成语法正确的示例 JSON，并在注释中注明 `[AI 推断]`。
- **最低保障**：对于每个端点，至少输出：
  - 请求方式与 URL
  - 请求头（若代码中有认证中间件，写 `Authorization: Bearer <token>`，否则留空）
  - 请求体结构（键名从模型或解析逻辑中提取）
  - 响应体结构（成功时返回的模型字段 + 常见错误码）
- **不生成无依据的示例**：如果连路由参数都未定义，则跳过该端点，标记 `[代码缺失：无法确定请求格式]`。

### 步骤 4：待验证清单与后续建议
在文档末尾附加一个“待验证清单”，列出 AI 推理但未确认的假设，以及活跃度分析中标记为“疑似废弃”的代码，建议后续通过用户访谈或数据分析验证。每个条目附带优先级（高/中/低），优先级根据置信度评分自动设置（置信度低的章节对应条目优先级为高）。

示例：
- `[高优先级] [AI 推断] 积分系统可能用于提升留存，但未在代码中找到实际兑换商城。是否存在未上线的商城功能？`
- `[中优先级] [代码实证：活跃度低] 路由 /api/v1/old/report 自 2021 年后无 commit，且无任何调用。建议确认是否可以删除。`
