---
name: produce-prd
description: 基于产品设计方法论，通过层层深入的对话，输出一份覆盖市场、需求、执行与风险的完整产品策略与需求文档（PRD），特别强化了需求拆解的粒度、技术规范的完整性以及对 MVP 交互原型包的支持。当产品包含对外 API（如 RESTful API）时，PRD 必须为每个端点提供实际使用的代码示例。
---

## 前置条件
用户有一个产品想法或待解决的问题，愿意花费 15～30 分钟进行深度对话。

## 核心目标
本 SKILL 的目标是输出一份 **开发可执行、测试可验证、设计可参考** 的 PRD，为此进行了三项关键增强：
1. **Phase/Task 分层拆解**：将需求按开发阶段分组，每阶段包含明确的 Task、依赖关系和里程碑定义
2. **完整技术规范**：包含核心数据体系、前端结构、命名规范、URL 清单、API 设计、数据接口、场景矩阵、数据库模型、错误处理
3. **MVP 交互原型包支持**：生成一套信息架构页面包的规格说明，使用静态数据、无样式、纯链接跳转，用于快速呈现系统全貌

## 执行流程

整个流程按照你的产品设计框架分为 4 个阶段。每阶段内按需发问，且允许用户随时回溯修正。

---

### 阶段 1：发现机会（为什么做）
目标：将模糊想法提炼为有数据支撑的、边界清晰的产品机会。

1. **问题与背景**
   - 用 `AskUserQuestion` 引导用户描述：当前遇到的/观察到的核心问题是什么？为什么是现在？目前用户如何解决（现有方案）？
2. **市场与行业速览**（如果用户无法回答，可主动触发一次简短的 WebSearch）
   - 要求用户提供或确认：目标市场大致规模、增长趋势、已有竞品或替代品。
   - 追问：现有竞品/替代品的最大缺陷是什么？
3. **目标用户画像**
   - 让用户用具体行为描述目标用户，而非抽象标签（例如"经常在深夜睡不着刷手机的上班族"而非"25-35岁失眠人群"）。
   - 追问：他们的核心痛点、现有解决方式、为之付费的意愿。
4. **产品逻辑（一句话提炼）**
   - 引导用户写下：`为【谁】解决【什么问题】，通过【什么方式】实现【什么价值】，并以【什么方式】变现`。
   - 这一步产出整个 PRD 的"北极星"。
5. **[新增] 核心成功指标定义**
   - 引导用户明确：**北极星指标**（能反映产品长期价值的关键指标，如 DAU、订单量、学习完成率等）；
   - **关键漏斗转化率**（从获客到核心转化的每一步，如访问→注册→首次使用→付费）；
   - **留存指标**（次日/7日/30日留存）；
   - **反向指标**（防止追求指标损害体验，如客服投诉率、功能报错率）。
   - 追问：如何验证产品是否成功？哪些数据变化会让你确信走在正确方向上？
   - 产出：成功指标清单（含定义、计算方式、目标值草案）。

**阶段 1 中间产出**：问题定义、市场简报、用户画像草稿、产品逻辑一句话、核心成功指标草案。

---

### 阶段 2：需求拆解（怎么做）
目标：将产品逻辑落成可开发的需求描述，覆盖用例、功能、数据、交互、Phase 拆解。

1. **解决方案概要**
   - 让用户用一段话说明其解决方案，并明确其商业模型（谁付钱、付多少、为什么付）。

2. **典型用户用例（User Cases）**
   - 至少产出 3 个端到端用户故事，每个包含：触发情景 → 操作步骤 → 期望结果。
   - 指导用户覆盖：首次使用、核心高频场景、异常/边缘场景。
   - **新增要求：用例编号化**，格式为 `UC-{序号} {场景名}`，便于后续 Task 引用。

3. **功能需求展开与优先级分层**  
   - 对每个用户故事，拆出具体功能点，并用 **MoSCoW** 标记优先级。
   - **新增要求：将 Must/Should 功能按 Phase 分组**
     - Phase 1：地基（如认证、路由、基础布局）— 无 Phase 依赖
     - Phase 2：骨架（如核心实体 CRUD、基础看板）— 依赖 Phase 1
     - Phase 3：血肉（如复杂交互、筛选、状态流转）— 依赖 Phase 2
     - Phase 4：闭环（如审核、通知、报告）— 依赖 Phase 3
     - 每个 Phase 包含 3~8 个代表性 Task
     - 每个 Task 标注：优先级、依赖、预估工时、**关联的 UC 编号（可选）**
     - 明确标识第几个 Phase 为 MVP 里程碑
   - **若产品提供对外 API（如 RESTful API），需在功能需求中明确标注“API 端点设计”为 Must 功能，并提示后续在技术规范中必须提供实际示例。**

4. **数据模型抽象**
   - 引导用户识别核心"名词"（实体），如用户、订单、内容、积分等，定义基本字段和关系。
   - **新增要求：输出 ER 图概要**（用文本描述实体关系和关键字段）
   - 如果用户不擅长，给出示例并帮助提炼。

5. **交互模型设计（八角行为分析驱动）**
   - 询问用户希望用户在使用产品时获得哪些**核心驱动力**（从八角行为分析框架选取）：
     - 史诗意义与使命感、进步与成就感、创意与反馈、所有权与拥有感、社交与联系、稀缺与渴望、未知与好奇、亏损与避免。
   - 基于选定的 2~3 个驱动力，共同设计关键的游戏化/行为激励机制，并反映在用例或功能描述中。

6. **场景矩阵**（新增）
   - 对于核心页面，建立场景矩阵：
     - 用户角色 × 页面状态 × 数据状态
     - 例如：列表页：已登录/未登录 × 有数据/无数据/加载中/加载失败 × 不同审核状态
   - 指导用户覆盖：首次使用（空状态）、加载中、加载失败、边界情况

**阶段 2 中间产出**：解决方案陈述、用例集（编号化）、功能清单+Phase/Task 分组+里程碑、ER 图概要、核心驱动力与交互原则、场景矩阵草稿。

---

### 阶段 3：资源与执行
目标：回归现实，评估团队/个人能否以合理成本实现。

1. **自身优势与已有资源**
   - 追问用户拥有的：技术栈能力、行业知识、人脉、种子用户、内容/IP、资金等。
2. **成本核算**
   - 粗略估算人力（前端/后端/设计/运营）、第三方服务（云、API）、营销预算。
   - 提示可"跳过"的环节（如早期用无代码搭建、不做安卓版本等）。
3. **执行路径与里程碑**
   - 与用户共同拟定 MVP 范围（通常是 Must + 少量 Should），设定 1~3 个月的里程碑目标。
   - **新增：基于 Phase 分组的里程碑定义，每个里程碑明确包含的 Phase 和交付物**
4. **营销与冷启动**
   - 头脑风暴：是否能利用已有资源在 2 周内获取第一批目标用户？最可行的低成本获客渠道是什么？

**阶段 3 中间产出**：资源清单、MVP 范围（按 Phase 定义）、里程碑计划、种子用户获取预案。

---

### 阶段 4：风险与挑战
目标：提前暴露可能让产品死亡的风险。

1. **市场与竞争风险**
   - 现有巨头或新进入者吞噬份额的可能性。
2. **执行风险**
   - 团队缺失关键能力、依赖第三方接口不稳定等。
3. **需求伪命题风险**
   - 用户不愿意付费或留存很低怎么办？设计什么样的验证实验？
4. **政策合规风险**
   - 数据隐私、行业准入门槛等。

---

### 最终生成 PRD 文档
对话结束后，将所有阶段性产出整合，输出到 `./docs/prd-{产品名}-YYYYMMDD.md`：

```markdown
# 产品策略与需求文档：{产品名}

> **文档版本**：v1.0
> **生成日期**：YYYY-MM-DD
> **状态**：草案

## 1. 发现机会
- 1.1 问题与背景
- 1.2 市场与行业速览
- 1.3 目标用户画像
- 1.4 产品逻辑（一句话北极星）
- 1.5 [新增] 核心成功指标
  - 1.5.1 北极星指标（名称、定义、计算方式）
  - 1.5.2 关键漏斗转化率（各阶段定义、目标值）
  - 1.5.3 留存指标（次日/7日/30日留存目标）
  - 1.5.4 反向指标（避免优化陷阱）

## 2. 需求拆解
- 2.1 解决方案与商业模式
- 2.2 典型用户用例（UC-01, UC-02, ...）
- 2.3 功能需求与 Phase 分组
  - 2.3.1 Phase 1：地基（Task 列表、依赖、里程碑）
  - 2.3.2 Phase 2：骨架
  - 2.3.3 Phase 3：血肉
  - 2.3.4 Phase 4：闭环
  - 2.3.5 完整 Task 清单（含优先级、依赖、预估工时、关联 UC）
- 2.4 数据模型摘要
  - 2.4.1 ER 图概要（实体关系文本描述）
  - 2.4.2 核心实体字段定义
- 2.5 场景矩阵
  - 2.5.1 列表页场景矩阵
  - 2.5.2 详情页场景矩阵
  - 2.5.3 表单页场景矩阵
- 2.6 交互模型（八角行为驱动设计，核心驱动力与机制）

## 3. 技术规范
- 3.1 技术栈总览
- 3.2 项目核心数据体系
  - 3.2.1 核心实体关系图（完整 ER 图描述）
  - 3.2.2 状态枚举定义（审核状态、转化状态等）
  - 3.2.3 业务规则清单（如收益分配比例、扣税规则）
- 3.3 核心功能定义
  - 3.3.1 列表筛选规范（每个筛选项：字段、类型、查询方式、选项来源）
  - 3.3.2 批量操作规范（可选数据范围、二次确认规则）
  - 3.3.3 详情页标签页结构
  - 3.3.4 审核/状态流转规则（状态图文本描述）
- 3.4 前端文件夹结构规范
  - 3.4.1 完整目录树（pages/assets/components/data）
  - 3.4.2 文件命名规范（页面、组件、数据文件）
  - 3.4.3 组件复用规则
- 3.5 数据命名规范
  - 3.5.1 URL 路径命名（kebab-case）
  - 3.5.2 CSS 类名（BEM 或项目约定）
  - 3.5.3 JavaScript 变量/函数（camelCase）
  - 3.5.4 数据库表/字段（snake_case）
  - 3.5.5 Git 分支/提交信息格式
- 3.6 前端页面/URL 清单
  - 3.6.1 完整路由表（路径、页面名称、所需数据、参数、对应组件）
  - 3.6.2 每个页面的 UI 状态分解（正常/空/加载/错误/权限不足）
- 3.7 API 设计
  - 3.7.1 完整端点清单（方法、路径、请求体、响应体、状态码）
  - 3.7.2 认证与鉴权机制
  - 3.7.3 错误码与错误消息对照表
  - 3.7.4 分页、排序、筛选参数规范
  - **3.7.5 实际使用示例（代码块）** — **强制要求**：为每个对外暴露的 RESTful 端点提供至少一组请求/响应示例，以代码块形式呈现。示例应包含：
    - 请求方式与 URL
    - 请求头（如 Authorization）
    - 请求体（JSON/XML/Form-Data）
    - 响应体（成功与失败场景各选典型）
    - 可选：cURL 命令或具体语言调用示例（如 JavaScript fetch、Python requests）
    - 示例格式例如：
      ````http
      POST /api/v1/orders
      Content-Type: application/json
      Authorization: Bearer {token}

      {
        "product_id": "123",
        "quantity": 2
      }
      ````
      ````json
      // 成功响应 (200 OK)
      {
        "code": 200,
        "data": {
          "order_id": "ord_789",
          "status": "created"
        }
      }
      ````
      ````json
      // 失败响应 (400 Bad Request)
      {
        "code": 400,
        "message": "库存不足"
      }
      ````
    - **注意**：若产品不包含对外 API，此节可注明“无对外 API”，跳过示例。
- 3.8 数据接口规范
  - 3.8.1 统一响应格式
  - 3.8.2 列表数据格式
  - 3.8.3 详情数据格式
  - 3.8.4 枚举值映射表
- 3.9 场景矩阵
  - 3.9.1 列表页场景矩阵
    - 用户已登录/未登录 × 数据为空/有数据/筛选后为空 × 加载中/加载成功/加载失败
  - 3.9.2 表单页场景矩阵
    - 新建/编辑 × 必填项为空/格式错误/提交中/提交成功/提交失败
  - 3.9.3 详情页场景矩阵
    - 数据加载中/加载成功/加载失败 × 各标签页 × 各审核状态
- 3.10 数据库模型（详细）
  - 3.10.1 每张表的完整 DDL（字段、类型、约束、默认值、注释）
  - 3.10.2 索引设计
  - 3.10.3 外键与关联关系
  - 3.10.4 分区策略（如适用）
- 3.11 错误处理规范
  - 3.11.1 前端错误处理（网络错误、表单验证错误、权限错误）
  - 3.11.2 后端错误处理（参数校验、业务异常、数据库错误）
  - 3.11.3 用户友好的错误提示文案清单
- 3.12 [新增] 数据埋点规范
  - 3.12.1 埋点基本原则（自动采集 vs 手动埋点、隐私合规）
  - 3.12.2 埋点事件命名规范（`{对象}_{动作}_{属性}`，如 `button_click_signup`）
  - 3.12.3 必埋事件清单（页面浏览、关键点击、业务流程转化、错误与异常）
  - 3.12.4 埋点参数定义表（事件名称、触发时机、参数键值、数据类型、示例）
  - 3.12.5 埋点测试与验证方法（本地抓包、日志校验）

## 4. MVP 交互原型包生成指南
- 4.1 目的与使用场景
  - 此交互原型包用于：视觉呈现系统全貌、辅助设计师设计、辅助开发理解功能、辅助测试编写用例
  - 特点：纯静态 HTML、无数据库交互、无样式美化、仅靠信息排版和链接跳转
- 4.2 文件结构
  - 4.2.1 每个页面一个独立 HTML 文件
  - 4.2.2 共享的 JSON 数据文件目录（静态 mock 数据）
  - 4.2.3 页面之间使用 `<a href>` 硬链接跳转
- 4.3 页面包清单
  - 列出每个 Phase 需要生成的页面文件
  - 每个页面标注：文件名、URL 路径、包含的状态变体
- 4.4 数据 mock 清单
  - 每个 mock 数据文件：文件名、内容结构、被哪些页面引用
- 4.5 生成命令建议
  - 推荐使用 `vite` 或简单 `live-server` 快速预览

## 5. 资源与执行
- 5.1 已有资源与优势
- 5.2 MVP 范围与里程碑（按 Phase 定义）
- 5.3 成本估算与降本方案
- 5.4 营销与冷启动计划

## 6. 风险与挑战

- 6.2 需要验证的关键假设

## 7. 测试用例 [增强：增加需求追溯链]

  - 7.1.1 Phase 1 测试用例
