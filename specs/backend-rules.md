# 后端代码规范

## 核心原则：人类可读性优先

### 命名

- 函数、变量、类名必须自解释。禁止使用任何无意义的缩写（例如：`us` 代替 `userService`、`d` 代替 `data`）。
- 控制器方法名必须直接反映对应的 HTTP 动词：
  - 获取资源：`getXxx`, `listXxx`
  - 创建资源：`createXxx`, `addXxx`
  - 更新资源：`updateXxx`, `patchXxx`
  - 删除资源：`deleteXxx`, `removeXxx`
- 数据库查询函数必须以操作动词开头：
  - 查找：`findBy...`, `findOneBy...`, `findAllBy...`
  - 创建：`create...`
  - 更新：`update...`
  - 删除：`delete...`

### 结构

- 严格遵循分层架构：**Controller → Service → Repository**。禁止跨层调用（如 Controller 直接操作数据库）。
- 每个类/文件应保持单一职责。如果出现以下情况，应立即拆分：
  - 类中混杂了多个不相关的业务领域
  - 同一个类既处理 HTTP 请求又包含复杂业务逻辑又直接写 SQL

### 错误处理

- 每个外部调用（数据库查询、远程 API、文件 I/O、消息队列等）都必须有错误处理逻辑（try-catch 或返回错误类型）。
- 错误信息必须包含足够的上下文以便快速定位，格式为：`"操作名称 失败，原因: 具体错误，参数: 关键参数值"`。
  - ✅ 示例：`"创建订单失败，原因: 数据库连接超时，参数: userId=123"`
  - ❌ 示例：`"出错了"` 或 `"数据库错误"`
- 对外暴露的 API 错误响应应包含：`code`（机器可读错误码）、`message`（人类可读描述）、`timestamp`。

### 注释

- 任何涉及复杂算法、非显而易见业务规则、或外部接口契约的代码，必须写注释解释「为什么这么做」以及「边界条件是什么」。
- 每个 API 端点（Controller 方法）必须有一行注释，描述其功能、预期的输入格式、以及可能返回的错误。
- 禁止对显而易见的代码添加注释（如 `i++  // 增加 i`）。
